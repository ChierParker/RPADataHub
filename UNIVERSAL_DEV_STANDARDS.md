# 通用开发约束与架构准则

> 适用范围：个人项目、原型、CLI 工具、Web 应用、后端 API、全栈项目、自动化脚本。
>
> 使用方式：把本文件放在项目根目录，并在给 AI 或协作者的任务说明中加入：
>
> **请遵守项目根目录下的 `UNIVERSAL_DEV_STANDARDS.md` 进行开发、重构和代码审查。**

---

## 0. 约束等级

本文使用三个等级，避免把所有建议都变成僵硬规则：

- **MUST**：必须遵守。违反后可能造成安全、数据、部署或维护风险。
- **SHOULD**：默认应遵守。若项目规模、时间或原型性质不适合，可说明原因后暂缓。
- **MAY**：可选优化。适合长期维护、多人协作或产品化阶段。

如果规则冲突，优先级为：安全与数据完整性 > 用户可用性 > 可维护性 > 代码风格。

---

## 1. 项目分级

不同项目不应套同一强度的架构。

| 级别 | 场景 | 最低要求 |
|---|---|---|
| Prototype | 快速验证、一次性脚本、Demo | MUST 项全部满足；核心逻辑尽量可测试 |
| Tool | 本地工具、小团队内部使用 | MUST + 关键 SHOULD；运行产物隔离；基础测试 |
| Product | 长期维护、交付客户、多人协作 | MUST + SHOULD；清晰分层；任务/数据可恢复 |
| Service | 多用户、并发、高可用服务 | Product 基础上增加队列、持久化、监控、权限控制 |

AI 或开发者开始改造前，应先判断项目级别；不确定时按 **Tool** 处理。

## 2. 目录与运行产物隔离

### MUST

源码目录只应包含可审查、可复现、可提交的文件。运行时生成的数据、用户会话、缓存、日志和构建产物必须被 `.gitignore` 排除。

推荐模板：

```gitignore
# Dependencies
node_modules/
.venv/
venv/
__pycache__/
*.py[cod]

# Test / cache
.pytest_cache/
.mypy_cache/
.ruff_cache/
.coverage
htmlcov/

# Runtime data
logs/
output/
input/
uploads/
tmp/
temp/
cache/
debug/

# User/session data
profiles/
sessions/
cookies/
*.local.*
*.private.*

# Secrets
.env
.env.local
.env.*.local
settings.local.json

# Build artifacts
build/
dist/
*.spec
*.exe
.next/
out/

# Editor / OS
.vscode/
.idea/
.DS_Store
Thumbs.db
```

### SHOULD

运行数据建议统一放在 `data/`、`var/` 或用户数据目录下，例如：

```text
data/
  input/
  output/
  logs/
  cache/
  profiles/
```

浏览器 profile、cookie、session、SQLite 真实数据、日志等可能含敏感信息，不得提交或分享。

## 3. 模块职责与分层

### SHOULD

每个模块应有清晰主职责。判断标准不是“绝对不能有和字”，而是：

- 文件名能准确表达职责。
- 修改某个业务点时，影响范围可预期。
- 纯逻辑、IO、副作用、框架入口尽量分开。
- 单文件超过约 500-800 行时，应主动评估是否拆分。
- 单函数超过约 60-100 行时，应评估是否拆出子步骤或纯函数。

### 常见分层

后端/API：

```text
routes/controllers  ->  services/use_cases  ->  repositories/gateways  ->  models/schemas
```

爬虫/自动化工具：

```text
runner/task  ->  browser/client  ->  parser/extractor  ->  exporter/storage  ->  state/progress
```

前端：

```text
pages/routes  ->  components  ->  hooks/store  ->  api/client  ->  utils
```

CLI：

```text
commands  ->  use_cases  ->  core logic  ->  io/adapters  ->  config
```

### MUST

不要把以下职责长期混在同一个文件里：

- 框架入口 + 业务编排 + 数据访问 + 文件导出
- 页面/路由 + 复杂状态机 + 外部服务调用
- 浏览器控制 + DOM 解析 + 数据清洗 + 持久化
- 真实密钥、运行状态、业务逻辑、测试数据混放

## 4. 状态管理

### MUST

可变共享状态必须明确归属，不能散落为多个无保护的模块级变量。

多线程或异步场景下，共享状态必须有以下之一：

- 锁保护，例如 `Lock` / `RLock`
- 单线程事件循环内的明确所有者
- 队列或消息通道
- 数据库/持久化存储作为状态源

### SHOULD

原型可以使用单个全局状态对象，但应封装成 `TaskState`、`TaskManager`、`SessionStore` 等对象，而不是散落变量。

演进路径：

```text
Prototype: 单任务状态对象
Tool: TaskManager + lock + 明确生命周期
Product: task_id + SQLite/文件持久化 + 可恢复
Service: 队列 + Redis/数据库 + worker
```

### MUST

如果服务重启后状态丢失会造成用户数据丢失、重复扣费、重复发送、任务无法恢复，则状态必须持久化。

## 5. 配置、密钥与编码

### MUST

- 源码、Markdown、JSON、YAML、TOML 使用 UTF-8。
- 读写文本文件时显式声明 `encoding="utf-8"`，除非处理第三方特定编码。
- 真实 API Key、密码、token、cookie 不得进入源码或示例配置。
- 示例配置只能使用占位值，例如 `your-api-key`、`your-password`。
- 本地真实配置放入 `.env.local`、`settings.local.json` 或系统密钥管理工具，并加入 `.gitignore`。

### SHOULD

配置按来源分层：

```text
默认配置 < 项目配置文件 < 环境变量 < 启动参数/用户输入
```

配置加载失败时应给出可理解错误，不应静默吞掉关键配置错误。非关键配置可使用默认值。

## 6. 输入验证与安全

### MUST

所有来自用户、HTTP 请求、CLI 参数、上传文件、外部服务的数据，在使用前必须验证。

文件上传：

- 校验扩展名白名单。
- 校验文件大小上限。
- 保存时使用随机文件名或安全文件名。
- 不允许用户输入直接拼接成保存路径。
- 解压文件必须防止 zip slip/path traversal。

路径处理：

- 用户可控路径必须 resolve 后确认仍在允许目录内。
- 禁止信任 `../`、绝对路径、软链接跳转。

数据库：

- SQL 使用参数化查询。
- ORM 查询也不得拼接不可信字符串作为原始 SQL。

Web/API：

- 后端必须重新校验前端已经校验过的参数。
- 错误响应不得泄露密钥、内部路径、完整堆栈或 cookie。

## 7. 可测试性

### SHOULD

核心业务逻辑优先写成纯函数或低副作用函数：

```python
def extract_emails(text: str) -> list[str]:
    ...
```

避免把核心逻辑写成只能依赖浏览器、数据库、全局变量或 UI 的形式。

测试分层：

| 层级 | 目标 | 示例 |
|---|---|---|
| Unit | 纯逻辑、数据清洗、格式转换 | 正则提取、去重、列名映射 |
| Integration | 文件读写、数据库、外部适配器 | Excel 导入导出、SQLite repository |
| E2E | 用户关键路径 | 上传文件、启动任务、下载结果 |

### SHOULD

不要机械追求覆盖率数字。最低要求是：

- 复杂纯逻辑有单元测试。
- 曾经出 bug 的逻辑有回归测试。
- 高风险 IO 或导出逻辑有烟测。
- 无法测试的代码应说明原因，并尽量隔离在适配器层。

## 8. 日志与可观测性

### MUST

- 生产代码不得依赖 `print()` 作为主要日志。
- 日志必须区分级别：`DEBUG`、`INFO`、`WARNING`、`ERROR`。
- 日志文件必须轮转，避免无限增长。
- 日志不得记录密码、token、cookie、完整身份凭证。

### SHOULD

关键流程日志应包含：

- 时间
- 模块名
- 任务 ID / trace ID
- 关键状态变化
- 可恢复错误的上下文

日志级别建议：

| 级别 | 使用场景 |
|---|---|
| DEBUG | 调试细节、请求参数摘要 |
| INFO | 正常流程节点，如任务开始、完成、导出成功 |
| WARNING | 可恢复异常，如重试、跳过、超时 |
| ERROR | 不可恢复异常或用户请求失败 |

## 9. API 与接口设计

### SHOULD

同一项目内 API 响应风格应一致。推荐 JSON 信封：

```json
{
  "success": true,
  "data": {},
  "error": ""
}
```

如果项目使用 REST、GraphQL、JSON:API、tRPC 等已有规范，以项目规范为准，但必须保持一致。

常用 HTTP 状态码：

| 场景 | 状态码 |
|---|---|
| 成功 | 200 |
| 创建成功 | 201 |
| 参数错误 | 400 |
| 未认证 | 401 |
| 无权限 | 403 |
| 不存在 | 404 |
| 冲突 | 409 |
| 限流 | 429 |
| 服务端错误 | 500 |

列表接口建议包含：

```json
{
  "items": [],
  "total": 0,
  "page": 1,
  "page_size": 20
}
```

## 10. 前端体验基线

### MUST

异步 UI 至少覆盖：

- 初始状态
- 加载状态
- 成功状态
- 空状态
- 错误状态

### SHOULD

- 表单字段有明确校验和错误提示。
- 危险操作有确认或撤销机制。
- 按钮在提交期间禁用或显示进度，避免重复提交。
- 静态资源优先本地化；若使用 CDN，应有可接受的降级策略。
- 页面不要依赖用户打开控制台才能理解错误。

## 11. 依赖与构建

### MUST

- 依赖文件必须存在并可复现，例如 `requirements.txt`、`pyproject.toml`、`package-lock.json`、`pnpm-lock.yaml`。
- 不要引入未使用的大型依赖。
- 构建产物不应提交，除非项目明确要求分发二进制。

### SHOULD

- 新增依赖前说明用途。
- 优先使用项目已有技术栈。
- 对安全敏感依赖保持可升级路径。
- README 中写清安装、启动、测试命令。

## 12. AI 协作规则

给 AI 分派任务时，默认要求：

- 先阅读项目结构和相关文件，再修改。
- 不重构无关代码。
- 不删除用户数据、运行产物或未确认的文件。
- 修改前说明将改哪些区域。
- 修改后运行可用的检查：编译、测试、lint 或烟测。
- 如果无法运行检查，必须说明原因。
- 对于架构优化，优先做小步、可验证、可回滚的改动。

AI 代码审查清单：

### P0: 必须修复

- [ ] 真实密钥、cookie、token、浏览器 profile 入库。
- [ ] 用户输入可导致路径遍历、SQL 注入、命令注入。
- [ ] 并发共享状态无保护且会造成数据错乱。
- [ ] 运行产物、日志、缓存污染源码目录且未忽略。
- [ ] 编码损坏导致配置、列名、用户文案不可用。
- [ ] 核心流程无法启动或无法编译。

### P1: 本轮应修复

- [ ] 单文件承担多个核心职责，已经影响修改安全性。
- [ ] 关键业务逻辑无法独立测试。
- [ ] 文件上传缺少扩展名、大小或路径校验。
- [ ] 错误处理只吞异常，没有日志或用户反馈。
- [ ] 魔法数字/字符串影响理解，应提取为命名常量。

### P2: 后续优化

- [ ] 文件或函数过长，建议拆分。
- [ ] API 响应不统一。
- [ ] 缺少 README 启动/测试说明。
- [ ] 缺少基础单元测试。
- [ ] 任务状态不可恢复，但当前阶段尚可接受。

## 13. 新项目初始化清单

```text
1. 创建 README，写明安装、启动、测试、构建命令。
2. 添加 .gitignore，排除依赖、运行产物、密钥、本地配置。
3. 建立 src/app 与 tests 目录。
4. 明确配置加载方式和本地配置文件命名。
5. 确认所有文本文件为 UTF-8。
6. 写一个最小可运行入口。
7. 写一个最小测试或烟测。
8. 让 AI 或协作者按本文件做第一次架构审查。
```

## 14. 项目适配说明

本文件是通用基线，不应压过项目实际情况。允许在项目根目录新增：

```text
PROJECT_DEV_STANDARDS.md
```

用于覆盖或补充本文件，例如：

- 当前项目技术栈
- 目录结构约定
- 命名规范
- API 响应格式
- 测试命令
- 部署方式
- 特定安全边界

覆盖规则：项目专属规范优先，但不得降低本文 P0 安全要求。
